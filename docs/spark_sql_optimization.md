# Spark SQL Optimization Notes

Milestone 4 focuses on reducing Spark SQL overhead without changing the public
output schemas. The representative inputs for physical-plan inspection and
before/after timing are `1m`, `full`, and `14m`.

Generate formatted physical plans with:

```powershell
.\.venv\Scripts\python.exe scripts\inspect_spark_sql_plans.py
```

The generated plan text is written under `outputs/spark_sql/plans/`, which is a
runtime output directory and remains untracked.

## Plan Observations

- The delay and all-causes jobs still scan the Parquet input, derive the same
  delay-range population, and shuffle for grouped aggregations.
- The previous top-three cause implementation used a grouped cause count
  followed by a window sort over cause rows. The optimized query keeps the
  grouped count but ranks the compact per-group cause list with sorted
  collected structs.
- The airline-airport ranking keeps its `RANK()` window because it runs after
  aggregation on the small airport-airline result and directly expresses the
  assignment semantics.
- Final output ordering remains deterministic, but Spark SQL now materializes
  each small aggregate result once and writes `full/part-00000.csv` plus
  `first_10.csv` from that same collected table, matching Spark Core's local
  output path.

## Chosen Optimization

The implementation does not cache the full input across jobs. Each benchmarked
job remains independently timed, and broad persistence would make those timings
less clear. Instead, the optimization removes repeated final actions inside a
job and replaces window-based cause ranking with compact grouped ranking.

`report/tables/spark_sql_optimization_before_after.md` is generated by
`make charts` when both baseline and optimized M4 benchmark rows are available.
It compares only these environment labels:

- `local-spark-sql-baseline-m4`
- `local-spark-sql-optimized-m4`
- `docker-simulation-spark-sql-baseline-m4`
- `docker-simulation-spark-sql-optimized-m4`
